Nora — Claude Instructions

System reference for Claude Code scheduled sessions and integrations

System Overview

Nora is a voice AI PM agent for LimeLight Marketing. She joins Zoom meetings via Recall.ai, listens to conversation through real-time transcription, responds when triggered by wake word ("Nora"), and speaks back using ElevenLabs TTS. She has persistent memory and a task queue that external agents (like Claude Code) can process.

Architecture

  • Server: Node.js/Express on Railway
  • Voice pipeline: Zoom → Recall.ai transcription webhook → Wake word detection → Claude Haiku → ElevenLabs TTS → Recall.ai output audio → Zoom
  • AI: Claude Haiku 4.5 for voice responses, Claude Haiku 4.5 for memory/task extraction
  • Storage: JSON files on Railway Volume (/data/) — memory and tasks persist across deploys
  • Dashboard: Web UI at root URL for meeting joins, task management, memory management

Wake Word & Triggers

  • Voice: Say "Hey Nora" or "Nora" followed by a question in a Zoom meeting
  • Chat: Type @nora your question in Zoom chat
  • Stop phrases: "Stop Nora", "Nora stop", "hold on", "never mind" — cuts off mid-speech

Authentication

The following endpoints require an API key: /memory, /projects, /tasks, /teamwork, /notify, and /transcripts. All other endpoints (dashboard, webhooks, join, mute, etc.) are open.

Pass the API key in one of two ways:

  • Query parameter: ?key=YOUR_NORA_API_KEY — simplest for GET requests and tools
  • Header: Authorization: Bearer YOUR_NORA_API_KEY

If NORA_API_KEY is not set in the environment, auth is disabled (open access for local dev).

// Examples:
GET /tasks?status=pending&key=YOUR_KEY
GET /memory?key=YOUR_KEY
GET /teamwork/tasks/12345/stage?stage=Done&key=YOUR_KEY
POST /notify  (with header: Authorization: Bearer YOUR_KEY)

Memory System

Nora has persistent long-term memory stored as a JSON array. Each memory is a fact string with metadata. Memory is appended to the system prompt on every Claude call so Nora always has access to everything she's been told.

Auto-extraction: After each voice response, a background Haiku call checks if someone explicitly asked Nora to remember something. Only explicit "remember this" requests and action items with clear owners/deadlines are saved.

Storage: /data/nora-memory.json on Railway Volume (falls back to nora-memory.json locally)

Projects

Projects give Nora deeper context on specific initiatives. Each project has a name and a set of structured fields (client, status, PM, phase, tags) plus free-text details. Memories tagged with a project name are grouped under it in Nora's system prompt. When a memory references a project that doesn't yet exist, the server auto-creates a stub record (marked auto_created: true) so /projects and /memory never drift apart.

Storage: /data/nora-projects.json on Railway Volume

Task Queue

Nora's task queue is the bridge between meetings and action. When someone asks Nora to do something specific in a meeting (schedule a meeting, send an email, create a Teamwork task), it gets extracted and added to the queue as a pending task.

For Claude Code scheduled sessions: Fetch pending tasks, execute them using MCP tools (Teamwork, Slack, Gmail, Google Calendar), then mark each one complete by ID. The complete endpoint is idempotent — calling it twice won't double-do anything.

Storage: /data/nora-tasks.json on Railway Volume

Claude Code Scheduled Session Workflow

This is the recommended workflow for a scheduled Claude Code session that processes Nora's task queue:

1. Fetch pending tasks:
   GET {BASE_URL}/tasks?status=pending

2. For each pending task, determine the right action:
   - "Schedule a meeting..." → use Google Calendar MCP to create event
   - "Send an email to..." → use Gmail MCP to draft/send
   - "Create a task in Teamwork..." → use Teamwork MCP to create task
   - "Send a Slack message..." → use Slack MCP to post message
   - "Remind [person] about..." → determine best channel and notify

3. Execute the action using the appropriate MCP tool

4. Notify the requester that it's done (posts as Nora):
   POST {BASE_URL}/notify
   {
     "channel": "C0123ABCDEF",   // from task.source_channel (strip "slack:" prefix)
     "user": "U0123ABCDEF",      // from task.source_user (used for DMs)
     "text": "Done — scheduled the follow-up with Kyle for Tuesday at 2pm."
   }
   Use "channel" for channel replies, "user" for DMs.
   If source_channel starts with "slack:", strip the prefix to get the channel ID.
   If source_channel is "zoom", skip Slack notify or DM the source_user instead.

5. Mark the task as done:
   PATCH {BASE_URL}/tasks/{task_id}/complete

6. Optionally, add a memory about what was done:
   POST {BASE_URL}/memory
   { "fact": "Sent Q2 report to Brandee on 2026-03-14", "source": "auto" }

Available MCP Integrations

  • Teamwork (twprojects): Create/update tasks, milestones, projects, time logs
  • Teamwork Desk (twdesk): Manage support tickets, customers, messages
  • Google Calendar (gcal): Create/update events, find free time, check availability

Slack Conversations

Nora supports real back-and-forth conversations in Slack — not just one-shot @mention replies. She responds to:

  • Every message in a DM
  • @mentions in channels (replies in a new thread)
  • Follow-up messages in any thread she has joined — no re-mention required, until the thread goes stale

Spam guards

Three layers protect against runaway chatter on thread continuation. None apply to DMs or explicit @mentions — those always respond.

  1. Auto-stale — a thread cools off after 5 messages without Nora being directly addressed, or 30 minutes since her last engagement. Stale threads require a re-mention to wake her back up.
  2. Heuristic skip — sub-4-character messages, emoji-only reactions, and messages mentioning another user (not Nora) are dropped before any LLM cost.
  3. Claude gate — every thread continuation passes through a cheap Haiku call asking "is this directed at Nora?" — defaults to no on uncertainty or error, so the failure mode is silence rather than spam.

Bridge to cowork (the routing fix)

The live Slack handler only has memory + prompt — no MCP tools. To keep conversations useful, when Nora hits a knowledge gap she queues a task and tells the user she'll follow up in this thread. Tasks queued from a Slack thread now carry source_thread_ts, which the cowork loop passes to /notify as thread_ts so the resolution lands in the original thread within the hour. From the user's side, the conversation is continuous; they don't need to know cowork ran.

Joined threads are persisted to /data/slack-threads.json so a deploy/restart doesn't drop active conversations. The list is capped at 1000 threads (oldest evicted).

Slack app config requirement: for thread continuation to work, the Slack app must subscribe to message.channels (and message.groups for private channels) in addition to app_mention and message.im. Without those, Slack only delivers explicit @mention events.

Proactive channel speaking (opt-in)

Nora can chime in on channel messages even without being @mentioned, when her live handler's stricter Claude gate decides she has substantive context to add. Strict opt-in per channel — default everywhere is off, because unsolicited interjections are a fast trust-breaker.

When enabled for a channel: every non-mention, non-thread message runs a stricter "is this a specific question Nora has specific context for?" Claude gate (defaults harder to "no" than the thread-continuation gate). The model gets a second chance to abort at generation time by returning empty. After each successful proactive post, the channel cools down for 30 minutes.

Teamwork Integration

Direct Teamwork API endpoints for operations not supported by the Teamwork MCP. These endpoints proxy requests to the Teamwork API using server-side credentials.

Move a Task to a Workflow Stage

Use this endpoint to move a Teamwork task to a different board/workflow stage (e.g. "To Do" → "In Progress" → "Done"). The Teamwork MCP (twprojects) does not support workflow stage changes, so use this endpoint instead.

GET /teamwork/tasks/:taskId/stage?stage=In+Progress
Move a Teamwork task to a different workflow stage by stage name (via query param)
// Example:
GET /teamwork/tasks/12345678/stage?stage=In+Progress

// Response:
{
  "ok": true,
  "taskId": "12345678",
  "stage": "In Progress",
  "workflowId": 98765,
  "stageId": 54321
}

How it works

  • Looks up the task to find its project ID
  • Fetches all workflows (boards) for that project
  • Searches each workflow's stages for a case-insensitive match on the stage name
  • Moves the task to the matched stage

Usage notes

  • The taskId is the Teamwork task ID (numeric), not a Nora task ID
  • Stage name matching is case-insensitive (e.g. "in progress" matches "In Progress")
  • If the project has multiple workflows/boards, all are searched until a matching stage is found
  • Returns 404 if the stage name doesn't exist in any workflow for the task's project

Example workflow

// 1. Find the Teamwork task ID (via Teamwork MCP or from a task's detail field)
const teamworkTaskId = "12345678";

// 2. Move it to a new stage
GET {BASE_URL}/teamwork/tasks/12345678/stage?stage=In+Progress

// 3. Common stage names: "Backlog", "To Do", "In Progress", "In Review", "Done"
//    (actual names depend on how the board is configured in Teamwork)
